  </read_first>
  <action>
    Clone the CalcpadCE community fork (NOT the discontinued Proektsoftbg/Calcpad — see RESEARCH.md Pitfall 1) and build a self-contained single-file Linux binary for Calcpad.Cli.

    Run exactly:
    ```bash
    rm -rf /tmp/CalcpadCE /tmp/calcpad-cli-out
    git clone https://github.com/imartincei/CalcpadCE.git /tmp/CalcpadCE
    cd /tmp/CalcpadCE
    dotnet publish Calcpad.Cli/Calcpad.Cli.csproj \
      -r linux-x64 --self-contained true \
      -p:PublishSingleFile=true -c Release \
      -o /tmp/calcpad-cli-out/
    ls -lh /tmp/calcpad-cli-out/
    file /tmp/calcpad-cli-out/Calcpad.Cli
    ```

    Expected:
    - Clone completes (no 404 — confirms RESEARCH.md is current)
    - `dotnet publish` exits 0
    - `/tmp/calcpad-cli-out/Calcpad.Cli` exists and `file` reports it as an ELF 64-bit executable
    - Binary size is multi-megabyte (self-contained bundles .NET runtime — typical 60–90 MB)

    If the publish fails due to a target framework mismatch (e.g., the csproj targets net8.0), capture the error and try `--framework net8.0` as a fallback. Record any deviation in the action result so plan 03 can document it.

    Per RESEARCH.md Open Question 2: confirm the actual binary name is `Calcpad.Cli` (no extension) by inspecting `/tmp/calcpad-cli-out/`. If the output filename differs, note it for the spike report.
  </action>
  <verify>
    <automated>test -x /tmp/calcpad-cli-out/Calcpad.Cli && file /tmp/calcpad-cli-out/Calcpad.Cli | grep -q 'ELF 64-bit'</automated>
  </verify>
  <acceptance_criteria>
    - `/tmp/CalcpadCE/.git/HEAD` exists (proves clone succeeded)
    - `/tmp/CalcpadCE/Calcpad.Cli/Program.cs` exists (proves expected source layout from research)
    - `/tmp/calcpad-cli-out/Calcpad.Cli` exists and is executable (`-x`)

    - Binary size > 10 MB (self-contained with bundled runtime)
  </acceptance_criteria>
