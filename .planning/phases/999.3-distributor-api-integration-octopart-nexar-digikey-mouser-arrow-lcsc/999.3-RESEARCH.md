## Summary

This phase adds real-time component inventory and pricing enrichment to LibreSpin's concept workflow by integrating six electronic component distributors via their public APIs. The integration is additive — the concept skill's Phase 4 gains an optional API enrichment step that fires when API keys are configured, and falls back gracefully to the existing WebFetch approach when they are not.

The six distributors split cleanly into three auth patterns: (1) **OAuth 2.0 client credentials** (DigiKey, Mouser) — curl POST to get a Bearer token valid for 10 minutes (DigiKey) or ~1 hour (Mouser); (2) **simple API key** (Nexar, Arrow, Newark/Farnell, LCSC) — API key passed as query parameter or header. No browser-redirect flow is required for any supplier. This is favorable for CLI/headless use.

The minimalism constraint (markdown skills, curl not libraries) is fully compatible with all six APIs. The most complex component is DigiKey's 10-minute token lifetime, which requires a refresh check before each API call. The INI credentials file at `~/.librespin/credentials` is the right storage model — it avoids gitignore risk entirely (global home dir, outside any project).

**Primary recommendation:** Build the setup skill first (validates each API works before saving keys), then patch concept Phase 4 to call a shell-based enrichment block for each configured supplier. All token refresh logic lives in the skill prompt as curl instructions — no Python or JS required.

## Authentication Patterns

### Pattern 1: Nexar OAuth 2.0 (client credentials)

**Token request:**
```bash
# Source: support.nexar.com/support/solutions/articles/101000434601
curl -s --request POST 'https://identity.nexar.com/connect/token' \
  --header 'Content-Type: application/x-www-form-urlencoded' \
  --data-urlencode 'grant_type=client_credentials' \
  --data-urlencode "client_id=$NEXAR_CLIENT_ID" \
  --data-urlencode "client_secret=$NEXAR_CLIENT_SECRET" \
  --data-urlencode 'scope=supply.domain'
# Returns: {"access_token":"...","expires_in":86400,"token_type":"Bearer"}
```

**GraphQL query (supSearchMpn):**
```bash
# Source: support.nexar.com (playground examples) + NexarDeveloper/nexar-examples-py
curl -s -X POST 'https://api.nexar.com/graphql/' \
  -H "Authorization: Bearer $NEXAR_TOKEN" \
  -H 'Content-Type: application/json' \
  -d '{
    "query": "query SearchMpn($q: String!) { supSearchMpn(q: $q, limit: 1) { results { part { mpn name manufacturer { name } specs(attribute: \"lifecyclestatus\") { displayValue } sellers(authorizedOnly: true) { company { name } offers { inventoryLevel moq prices { quantity price currency } } } } } } }",
    "variables": { "q": "LM358" }
  }'
```

**Data fields available:**
- `mpn` — Manufacturer Part Number
- `manufacturer.name` — Manufacturer name
- `specs` with `attribute: "lifecyclestatus"` — lifecycle status (e.g., "Production")
- `sellers.offers.inventoryLevel` — stock count at distributor
- `sellers.offers.prices` — price breaks `[{quantity, price, currency}]`
- `sellers.offers.moq` — minimum order quantity
- `sellers.offers.factoryLeadDays` — factory lead time (available in supParts, may require second query)

**Free tier limit:** 100 matched parts (total, not monthly reset). Detect exhaustion from error response and surface per D-18.

### Pattern 2: DigiKey OAuth 2.0 (client credentials)

**Token request:**
```bash
# Source: developer.digikey.com/tutorials-and-resources/oauth-20-2-legged-flow
curl -s -X POST 'https://api.digikey.com/v1/oauth2/token' \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  -d "client_id=$DK_CLIENT_ID&client_secret=$DK_CLIENT_SECRET&grant_type=client_credentials"
# Returns: {"access_token":"...","expires_in":599,"token_type":"Bearer"}
# Token valid for ~10 minutes. Must refresh before each API call if expired.
```

**Part search:**
```bash
# Source: developer.digikey.com/documentation
curl -s -X GET "https://api.digikey.com/products/v4/search/$MPN/productdetails" \
  -H "Authorization: Bearer $DK_TOKEN" \
  -H "X-DIGIKEY-Client-Id: $DK_CLIENT_ID" \
  -H "X-DIGIKEY-Locale-Site: US" \
  -H "X-DIGIKEY-Locale-Language: en" \
  -H "X-DIGIKEY-Locale-Currency: USD"
```

**Key detail:** `X-DIGIKEY-Client-Id` header is REQUIRED on all requests (not just token request). Missing it causes auth failure.

**Sandbox vs production:** Sandbox endpoint is `sandbox-api.digikey.com`. Same token endpoint structure but with `sandbox-api`. For validation in `/librespin:setup`, use sandbox if user has sandbox credentials — otherwise use production with a known safe MPN (e.g., `LM358`).

**Rate limits:** 120 requests/minute burst, ~1000 requests/day. Returns HTTP 429 with `Retry-After` header on limit hit.

### Pattern 3: Mouser API key

**Part search:**
```bash
# Source: github.com/sparkmicro/mouser-api, mouser.com API docs (confirmed pattern)
curl -s -X POST "https://api.mouser.com/api/v1/search/partnumber?apiKey=$MOUSER_API_KEY" \
  -H 'accept: application/json' \
  -H 'Content-Type: application/json' \
  -d "{\"SearchByPartRequest\": {\"mouserPartNumber\": \"$MPN\", \"partSearchOptions\": \"\"}}"
```

**Key detail:** Mouser has two separate API keys — `MOUSER_PART_API_KEY` (for search) and `MOUSER_ORDER_API_KEY` (for orders). This phase only needs the search key.

**D-04 correction:** Mouser uses API key, not OAuth 2.0. The credentials section `[mouser]` stores one key (`part_api_key`), no token refresh needed.

### Pattern 4: Arrow API key

**Part search:**
```bash
# Source: developers.arrow.com/api (Pricing & Availability API v4)
curl -s "https://api.arrow.com/itemservice/v4/en/search?term=$MPN&login=$ARROW_LOGIN&apikey=$ARROW_API_KEY"
```

**Key detail:** Both `login` (Arrow account email/username) and `apikey` are required. Arrow provides a request form to obtain API keys — not instant signup. Credentials section needs `[arrow]` with `login` and `api_key`.

**Availability note:** Arrow API access requires contacting api@arrow.com or using the "Request API Key" page. Not guaranteed to be frictionless for all users — setup skill must handle gracefully if user skips.

### Pattern 5: Newark/Farnell (element14 Partner Network)

**Part search:**
```bash
# Source: partner.element14.com/docs
curl -s "https://api.element14.com/catalog/sandboxed/products;keywordList=$MPN;storeInfo.id=uk.farnell.com/v1/xml-data/productDetails/manufacturer?apikey=$ELEMENT14_API_KEY"
```

**Key detail:** element14 API covers Farnell (UK/EU), Newark (US), and element14 (Asia). One API key covers all three storefronts. The `storeInfo.id` parameter selects the storefront: `uk.farnell.com`, `us.newark.com`, `sg.element14.com`. For US users, `us.newark.com` is the Newark storefront.

**Registration:** Self-service at `partner.element14.com` — register and apply for an API key. No enterprise agreement required for Product Search API.

**Response format:** XML by default; can request JSON with appropriate accept header.

### Pattern 6: LCSC API key

**Key finding:** LCSC has an official OpenAPI at `https://www.lcsc.com/docs/openapi/index.html`, but access requires: (1) an LCSC account, and (2) submitting an API access application. Not instant — users must apply. Rate limits: 1000 searches/day, 200/minute.

**Unofficial endpoint (documented by community):**
```bash
# Source: github.com/yaqwsx/jlcparts, github.com/Bouni (gist) — MEDIUM confidence
curl -s "https://wmsc.lcsc.com/wmsc/search/global?keyword=$MPN&currentPage=1&pageSize=10"
# No auth required for basic search — public endpoint used by LCSC website
```

**Note:** The community-documented `wmsc.lcsc.com` endpoint requires no auth and returns JSON. It is the endpoint the LCSC website itself uses. This is not an officially documented API and could change without notice. For setup validation: use this public endpoint. For production use: prefer official OpenAPI if user has credentials.

**Recommendation for skill:** Check for `[lcsc]` section in credentials. If `api_key` present, use official OpenAPI; if absent but LCSC is enabled (user opted in), use the public wmsc endpoint as fallback. Flag clearly: "LCSC official API requires application approval — using public endpoint as fallback."

**Bash parsing in skill (no INI library needed):**
```bash
# Read a value from credentials file
NEXAR_CLIENT_ID=$(grep -A10 '^\[nexar\]' ~/.librespin/credentials | grep 'client_id' | head -1 | awk -F' = ' '{print $2}')
# Token expiry check
EXPIRES=$(grep -A10 '^\[nexar\]' ~/.librespin/credentials | grep 'token_expires' | head -1 | awk -F' = ' '{print $2}')
NOW=$(date -u +%Y-%m-%dT%H:%M:%SZ)
# If EXPIRES <= NOW, refresh token
```

**Write-back for token caching:** The skill uses `sed -i` or `awk` to update token values in place within the credentials file. This is the standard shell pattern for INI file mutation without a parser library.

### Anti-Patterns to Avoid

- **Blocking concept on API failure:** Any API error must log and continue. Never throw or halt.
- **Hardcoding the Nexar parts count:** Read `parts_used` from credentials file; write it back after each query.
- **Assuming DigiKey token is valid:** 10-minute lifetime means check expiry before every request. The skill that checks the expiry and refreshes if needed must be in a Bash block, not prose.
- **Using Node.js or Python to parse INI:** Shell `grep` + `awk` is sufficient for this simple format. No library dependency introduced.
- **Ignoring the `X-DIGIKEY-Client-Id` header:** This header is mandatory on every DigiKey API request, not just the token request. Omitting it causes auth failure even with a valid Bearer token.
- **Treating LCSC wmsc endpoint as stable:** Flag it as unofficial in skill comments.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| OAuth 2.0 token exchange | Custom auth state machine | `curl` POST to token endpoint | One curl call returns the token; no library needed |
| INI file parsing | Custom INI parser in awk/sed | `grep -A10 '^\[section\]'` pattern | Sufficient for flat INI with simple k=v pairs |
| API response parsing | JSON parser in bash | `jq` (available on Linux) or constrained `grep`/`awk` | jq is standard on Linux; use for JSON extraction in Bash blocks |
| Rate limit back-off | Exponential back-off loop | Single retry after `Retry-After` header value | Only DigiKey/Mouser return Retry-After; one retry is enough, don't loop |
| Credential encryption | Custom encryption | Leave credentials in plaintext in `~/.librespin/` | Keys are low-sensitivity (read-only API access); encryption adds friction vs. minimal security gain; document clearly |

**Key insight:** The distributor API auth patterns are simple enough that pure curl handles them. The complexity budget should go into the skill prompt logic (token expiry detection, error handling, enrichment formatting), not into tooling.

## Common Pitfalls

### Pitfall 1: DigiKey's 10-Minute Token Lifetime
**What goes wrong:** Token cached from setup is already expired by first concept run. API returns 401.
**Why it happens:** Token lifetime is 599 seconds (~10 min) — shorter than any workflow session.
**How to avoid:** Before every DigiKey API call, compare stored `token_expires` to current UTC time. If expired (or within 60 seconds of expiry), refresh first. Write new token + new expiry to credentials file immediately.
**Warning signs:** HTTP 401 response from DigiKey API despite having a token.

### Pitfall 2: Nexar Free Tier Silent Exhaustion
**What goes wrong:** 100-part quota consumed without user awareness; subsequent queries silently fail or return quota error mid-workflow.
**Why it happens:** Each `supSearchMpn` call consumes quota. No built-in warning in the API response until limit is hit.
**How to avoid:** Track `parts_used` in `[nexar]` section of credentials file. Increment after each successful query. At 80 parts, warn user: "Nexar free tier: 80/100 parts used." At 100, surface the D-18 message and skip Nexar.
**Warning signs:** Nexar response includes a quota/limit error body.

### Pitfall 3: Missing X-DIGIKEY-Client-Id Header
**What goes wrong:** API returns 401 or 403 despite valid Bearer token.
**Why it happens:** DigiKey requires `X-DIGIKEY-Client-Id` as a header on every request, separate from the Authorization Bearer.
**How to avoid:** Always include both headers: `Authorization: Bearer TOKEN` AND `X-DIGIKEY-Client-Id: CLIENT_ID`.
**Warning signs:** 401/403 from DigiKey with a token that was just freshly obtained.

### Pitfall 4: Arrow Requires Both Login and API Key
**What goes wrong:** API key alone returns auth error; so does login alone.
**Why it happens:** Arrow's auth model is two-factor: account login (email) + api_key. Both are URI params.
**How to avoid:** Setup skill must prompt for both `login` (Arrow account email) and `api_key`. Store both in `[arrow]` section.
**Warning signs:** HTTP 401 from Arrow even with a valid API key.

### Pitfall 5: LCSC Official API Is Not Instant Access
**What goes wrong:** User enters setup, enables LCSC, but has no API key yet because application is pending.
**Why it happens:** LCSC requires submitting an application and awaiting approval — not instant.
**How to avoid:** In setup, LCSC section should offer two modes: (a) "I have an API key" → prompt for key, validate with official endpoint; (b) "I want to use public endpoint" → validate wmsc endpoint with test MPN, note it's unofficial.
**Warning signs:** User reports "LCSC returned invalid key" but they're actually in pending approval state.

### Pitfall 6: sed -i on macOS vs Linux
**What goes wrong:** `sed -i` with no extension works on Linux but fails on macOS (requires `sed -i ''`).
**Why it happens:** BSD sed (macOS) requires explicit empty string argument; GNU sed (Linux) does not.
**How to avoid:** Use `sed -i.bak` for cross-platform safety, then `rm *.bak`. Or use Python one-liner if sed portability is a concern. Since LibreSpin is Linux-primary (CalcPad CE spike was Linux), acceptable to document as Linux-only for v1.
**Warning signs:** Token write-back fails silently on macOS users.

### Pitfall 7: Nexar scope must be "supply.domain"
**What goes wrong:** Token obtained without scope, or with wrong scope — GraphQL supply queries return permission denied.
**Why it happens:** Nexar has separate scopes: `supply.domain` for Octopart/supply data, `design.domain` for Altium 365. Only `supply.domain` is needed here.
**How to avoid:** Always include `scope=supply.domain` in the token request body.
**Warning signs:** GraphQL response returns "permission denied" or empty results despite valid token.

## Code Examples

### Credentials File Read (shell — no parser library)

```bash
# Source: standard shell pattern, verified against bash 5.x
read_credential() {
  local section="$1" key="$2" file="$HOME/.librespin/credentials"
```

### Token Expiry Check (shell)

```bash
# Source: standard shell date comparison
is_token_expired() {
  local expires="$1"  # format: 2026-04-08T22:00:00Z
  local now
  now=$(date -u +%Y-%m-%dT%H:%M:%SZ)
  [[ "$now" > "$expires" ]]
}

EXPIRES=$(read_credential digikey token_expires)
if is_token_expired "$EXPIRES"; then
  # Refresh token
  RESPONSE=$(curl -s -X POST 'https://api.digikey.com/v1/oauth2/token' ...)
  NEW_TOKEN=$(echo "$RESPONSE" | jq -r '.access_token')
  NEW_EXPIRY=$(date -u -d "+9 minutes" +%Y-%m-%dT%H:%M:%SZ)  # 599s - 60s buffer
  # Write back to credentials file
  sed -i "s|^access_token = .*|access_token = $NEW_TOKEN|" ~/.librespin/credentials
  sed -i "s|^token_expires = .*|token_expires = $NEW_EXPIRY|" ~/.librespin/credentials
fi
```
