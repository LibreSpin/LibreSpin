# Phase 999.3: Distributor API Integration - Context

**Gathered:** 2026-04-08
**Status:** Ready for planning

<domain>
## Phase Boundary

Enable real-time component inventory and pricing verification via distributor APIs, replacing the current bot-blocked WebFetch approach. Deliver secure API key management (global credentials store) and automatic enrichment of component lookups during concept Phase 4 (component research).

**In scope:** All 6 suppliers (Nexar/Octopart, DigiKey, Mouser, Arrow, Newark/Farnell, LCSC), credential setup command, Phase 4 enrichment integration.

**Out of scope:** Standalone BOM verification skill (post-concept pass), batch re-query of existing .librespin outputs, scheduled price alerts.

</domain>

<decisions>
## Implementation Decisions

### Supplier Scope (D-01 through D-06)
- **D-01:** Implement all 6 suppliers in this phase: Nexar/Octopart, DigiKey, Mouser, Arrow, Newark/Farnell, LCSC.
- **D-02:** Nexar/Octopart — GraphQL API, API key auth. Free tier: 100 matched parts total — skill must surface this limit to the user and warn when approaching.
- **D-03:** DigiKey — OAuth 2.0 full implementation (client ID + secret, token exchange, token refresh). Researcher must investigate DigiKey sandbox vs production environments and token refresh cadence.
- **D-04:** Mouser — OAuth 2.0 full implementation. Same auth pattern as DigiKey; researcher to confirm if token refresh works identically or has differences.
- **D-05:** Arrow and Newark/Farnell — Researcher to determine auth type (API key vs OAuth vs none). Implement whatever the public API requires.
- **D-06:** LCSC — REST API, likely API key auth (researcher to confirm). Priority supplier for low-cost Chinese components — availability data is particularly valuable here.

### Credential Storage (D-07 through D-09)
- **D-07:** Global credentials file at `~/.librespin/credentials` — enter once, persists across all projects. No gitignore risk (outside any repo).
- **D-08:** File format: INI-style sections per supplier, e.g. `[nexar]`, `[digikey]`, `[mouser]`. Each section holds the keys/tokens that supplier requires. Human-readable.
- **D-09:** Token caching for OAuth suppliers: store access tokens + expiry in credentials file. Skill refreshes token automatically when expired, without re-prompting user.

### Setup Command (D-10 through D-11)
- **D-10:** Deliver a `/librespin:setup` skill (new skill file `skills/setup/SKILL.md`) that walks the user through key entry for each supplier and validates each API before saving. Matches the UX goal in ROADMAP.
- **D-11:** Setup is optional per supplier — user can skip any supplier they don't have an account for. Suppliers with no key configured are silently skipped during concept enrichment.

### Integration Point (D-12 through D-14)
- **D-12:** Enrichment lives inside the existing concept skill's Phase 4 (component research). When the concept agent selects specific components with MPNs, it calls distributor APIs to attach inventory count and price to each component inline.
- **D-13:** If no suppliers are configured (no credentials file or all empty), Phase 4 falls back to the existing behavior (WebFetch to manufacturer sites + "flag inventory as human-verify"). No breakage.
- **D-14:** Results appended to the component entry in `.librespin/05-components/` (or wherever Phase 4 currently writes). Do NOT change the output contract for downstream phases (CalcPad reads from `07-final-output/`, not component research files).

### Auth Complexity (D-15 through D-16)
- **D-15:** DigiKey and Mouser: implement full OAuth 2.0 — client credentials flow or authorization code flow depending on what their developer portal supports. Researcher must determine which grant type each supports for server-side/CLI use.
- **D-16:** OAuth token exchange and refresh are handled by the skill agent via `curl`. No OAuth library required — keep it pure shell/curl consistent with the minimalism constraint. If the flow requires a browser redirect (authorization code grant), researcher must identify an alternative (device flow or client credentials).

### Fallback and Error Handling (D-17 through D-18)
- **D-17:** If a supplier API call fails (rate limit, network error, bad key), log the failure inline and continue with remaining suppliers. Never block concept workflow progress on API failure.
- **D-18:** Nexar free tier exhaustion: detect "quota exceeded" response, surface a clear message to user ("Nexar free tier (100 parts) exhausted — upgrade or configure a different supplier"), then continue with other suppliers.

</decisions>

<specifics>
## Specific Ideas

- The setup command (`/librespin:setup`) should validate each API before saving — entry without validation is a footgun (typos in keys are silent failures mid-workflow).
- LCSC is specifically called out as the go-to for low-cost Chinese components — availability and pricing data from LCSC is particularly valuable for hobbyist/student target users.
- Nexar's 100-part free tier is a known constraint: the skill should make this visible to the user rather than silently consuming quota.
- UX goal from ROADMAP: "make key setup as frictionless as possible."

</specifics>
